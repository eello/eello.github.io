    데이터베이스 복제(Replication) 기술 총정리: 개념부터 아키텍처까지

    서비스의 규모가 커지면 단일 데이터베이스(DB) 서버만으로는 성능과 안정성을 보장할 수 없다. 이때 필수적으로 도입되는 기술이 바로 **데이터베이스 복제(Replication)**다. 복제 기술의 본질과 이를 활용한 다중화 전략을 정리해본다.

    1. 데이터베이스 복제의 본질

    데이터베이스 복제는 DB 서버 자체를 복사하는 기술이 아니라, '데이터의 변화(내용물)'를 복사하는 기술이다. 운영자가 미리 준비한 별도의 DB 서버(Slave)에 마스터(Master)의 변경 사항을 실시간으로 전달하여 동기화하는 것이 핵심이다.

    로그 기반의 동기화 프로세스

    복제는 마스터의 **바이너리 로그(Binary Log)**를 기반으로 움직인다.

    1. 기록: 마스터에서 데이터 변경 발생 시 이를 바이너리 로그에 기록한다.

    2. 전달: 슬레이브가 마스터의 로그를 요청하여 자신의 릴레이 로그(Relay Log)에 저장한다.

    3. 재현: 슬레이브가 로그 내용을 읽어 자신의 DB에 그대로 실행(Replay)한다.

    2. 동기화 방식에 따른 분류

    마스터가 슬레이브의 데이터 반영 상태를 어디까지 확인하느냐에 따라 복제 방식이 나뉜다. 이는 시스템의 데이터 정합성과 성능 사이의 트레이드오프(Trade-off)를 결정하는 중요한 요소다.

    비동기 방식 (Asynchronous)

    • 메커니즘: 마스터는 바이너리 로그에 변경 사항을 기록하고 슬레이브에 전달한 뒤, 슬레이브의 수신 여부와 상관없이 즉시 작업을 완료한다.

    • 장점: 마스터의 쓰기 성능이 가장 빠르다.

    • 단점: 마스터 장애 시 슬레이브로 전달되지 못한 데이터가 유실될 위험이 있다.

    동기 방식 (Synchronous)

    • 메커니즘: 마스터에 데이터가 기록되면 모든 슬레이브에도 데이터가 반영될 때까지 마스터가 대기한다. 모든 서버가 응답해야 비로소 트랜잭션이 완료된다.

    • 장점: 데이터 일관성이 완벽하게 보장된다.

    • 단점: 슬레이브 중 한 대만 느려져도 전체 시스템의 쓰기 성능이 급격히 저하된다.

    반동기 방식 (Semi-synchronous)

    • 메커니즘: 마스터가 로그를 보낸 후, 최소 한 대의 슬레이브가 로그를 받았다는 신호(ACK)를 보낼 때까지만 대기한다.

    • 장점: 비동기보다 안전하고 동기 방식보다 빠르다. 성능과 안정성의 절충안으로 실무에서 많이 사용된다.

    3. 복제 기반의 다중화 전략: Active-Standby vs Active-Active

    액티브-스탠바이 (Active-Standby)

    • 구조: 마스터(Active)는 모든 요청을 처리하고, 슬레이브(Standby)는 복제만 받으며 대기한다.

    • 특징: 장애 발생 시 스탠바이를 마스터로 **승격(Promotion)**시켜 서비스를 잇는다.

    읽기 분산형 액티브-액티브

    • 구조: 마스터는 쓰기/읽기를 수행하고, 슬레이브는 읽기(Read) 요청을 분담한다.

    • 특징: 내부적으로는 마스터-슬레이브 구조지만, 모든 서버가 일을 하고 있으므로 운영 관점에서는 액티브-액티브처럼 동작한다.

    4. 클러스터링과 프록시: 자동화의 완성

    단순 복제만으로는 장애 발생 시 사람이 개입해야 하는 수동 작업이 남는다. 이를 자동화하기 위해 클러스터링과 프록시 기술이 결합된다.

    • 클러스터링: 마스터 장애 시 자동으로 슬레이브를 승격시키는 페일오버(Failover)를 담당한다.

    • 프록시(ProxySQL 등): 쓰기는 마스터로, 읽기는 슬레이브로 보내는 Read-Write Splitting을 수행하며 접속 점을 단일화한다.

    5. 고급 아키텍처: 멀티 마스터와 계층형 복제

    • 멀티 마스터 (Active-Active): 모든 노드가 쓰기 권한을 갖는다. 양방향 복제가 일어나며 데이터 충돌 해결 로직이 필수적이다.

    • 계층형 복제: 멀티 마스터 하위에 각각 슬레이브들을 두는 구조다. 쓰기 부하를 분산하면서도 읽기 확장성을 무한히 늘릴 수 있다.

    6. 결론: 왜 다중화를 하는가?

    데이터베이스 다중화의 목적은 명확하다. 단일 장애점(SPOF)을 제거하여 **고가용성(HA)**을 확보하고, 부하 분산을 통해 성능을 최적화하며, 실시간 복제로 데이터를 안전하게 보호하는 것이다. 엔지니어는 서비스의 특성에 맞춰 적절한 동기화 방식과 아키텍처를 선택해야 한다.
