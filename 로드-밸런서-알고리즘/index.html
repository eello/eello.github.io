    로드 밸런서 알고리즘

    Published on 2026-01-19
    Category: infra > 로드 밸런서

    기초 알고리즘

    정적 알고리즘

    연결된 서버의 현재 상태를 고려하지 않고 정해진 규칙에 따라 요청을 분배

    라운드 로빈(Round Robin)

    • 요청을 순차적으로 서버에 할당하는 방식
    • 모든 서버의 사양이 동일하고 세션이 짧은 서비스에 적합

    가중치 라운드 로빈(Weighted Round Robin)

    • 각 서버에 가중치를 부여
    • 가중치가 높은 서버에 더 많은 요청 할당
    • 성능이 더 좋은 서버에 더 많은 요청을 할당할 때 사용

    IP 해시(IP Hash)

    • 클라이언트의 IP 주소를 해싱
    • 해싱된 IP 주소와 서버를 매핑
    • 사용자가 항상 동일한 서버로 연결되어야 하는 경우 사용

    동적 알고리즘

    서버의 실시간 상태(연결 수, 응답 시간 등)을 분석해 최적의 서버로 요청 할당

    최소 연결 방식(Least Connection)

    • 현재 연결(Session) 수가 가장 적은 서버로 요청 할당
    • 요청마다 처리 시간이 길어 서버 간의 부하 불균형이 생기기 쉬운 환경에 적합

    최소 응답 시간(Least Response Time)

    • 연결 수와 서버의 응답 시간을 모두 고려
    • 가장 빨리 응답할 수 있는 서버를 선택해 요청을 할당

    최소 대역폭(Least Bandwidth)

    • 최근 일정 시간 동안 가장 적은 대역폭(초당 전송량)을 사용 중인 서버에 할당

    심화 알고리즘

    실제 서비스나 복잡한 인프라 환경에서는 더 정교한 알고리즘 사용

    정교한 동적/적응형 알고리즘

    서버의 물리적 상태나 네트워크 패킷의 세부 흐름을 실시간으로 감지하여 대응

    최소 대기 방식(Adaptive / Least Pending Connection)

    • 연결 수만 세는 것이 아닌 응답이 완료되지 않은 요청을 고려
    • 응답이 완료되지 않은(Pending) 요청이 가장 적은 서버로 분배
    • 트래픽이 폭증할 때 병목 현상을 가장 빠르게 감지

    리소스 기반 알고리즘(Resource-Based / Agent-Based)

    • 서버에 설치된 에이전트가 CPU 사용률, 메모리 잔량 등의 정보를 로드 밸런서에 전달
    • 로드 밸런서는 전달받은 지표를 보고 가장 여유 있는 서버로 분배

    최소 패킷 방식(Least Packets)

    • 일정 시간 동안 처리한 패킷 수가 가장 적은 서버에 분배
    • 데이터 전송량이 서버 부하의 핵심인 서비스에 유리

    L7(Application Layer) 기반 알고리즘

    애플리케이션 계층의 정보를 읽어 '내용’에 따라 분배

    URL 해시(URL Hash)

    • 사용자가 요청하는 URL을 해싱하여 특정 서버로 분배
    • 특정 이미지나 비디오 콘텐츠가 특정 서버에만 캐싱되었을 때 효율적

    쿠키 기반 알고리즘(Cookie Insertion / Sticky Session)

    • 클라이언트의 쿠키 정보를 활용
    • 쿠키 정보를 확인해 이전에 접속했던 서버로 계속 연결
    • 장바구니 내역처럼 서버 세션 유지가 필수인 쇼핑몰 등에서 필수적

    HTTP 헤더/페이로드 분석

    • 요청 헤더를 활용
    • 헤더에 담긴 기기 정보(Mobile/PC)나 언어 설정에 따라 서로 다른 서버 팜(Server Farm)으로 트래픽 분산
    서버 팜(Server Farm)
    여러 대의 서버를 한곳에 모아 네트워크로 연결해 하나의 거대한 시스템처럼 작동하게 만든 서버 집합체

    구조

    • 로드 밸런서(Load Balancer): 서버 팜의 ‘입구’ 역할, 외부에서 들어오는 요청을 팜 내부 여러 서버들에게 분배
    • 컴퓨팅 노드(Individual Servers): 데이터를 처리하는 개별 서버
    • 중앙 관리 시스템: 수많은 서버의 상태를 모니터링하고, 소프트웨어 업데이트나 보안 패치를 한꺼번에 적용

    확률 및 기타 방식

    랜덤 방식(Random)

    • 무작위 서버로 분배
    • 매우 단순
    • 서버 성능이 모두 동일하고 요청 수가 매우 많을 때는 통계적으로 고르게 분산되는 효과

    최단 거리 방식(Geographic / Static Proximity)

    • 사용자의 물리적 위치(IP 기반)와 가장 가까운 곳의 데이터센터 혹은 서버로 연ㄷ결
    • 글로벌 서비스에서 응답 속도를 높이기 위해 사용

    Sticky Session 방식 비교: IP 해시 vs 쿠키 기반

    두 방식 모두 특정 사용자를 특정 서버에 고정한다는 점에서 동일하다.
    하지만 무엇을 기준으로, 네트워크 환경 변화에 대한 유연성에서 큰 차이가 있다.

    IP 해시

    • 별도의 쿠키 설정이 필요 없어 설정이 매우 간단
    • HTTP 뿐만 아니라 TCP 수준의 로드 밸런싱에서도 사용 가능

    하지만 치명적 단점이 존재한다.

    • 공유 IP 문제
      사내 망이나 공공 와이파이처럼 수백 명이 하나의 공인 IP를 공유하는 경우 모든 사용자가 단 하나의 서버로 고정
    • 환경 변화
      사용자가 이동 중에 와이파이에서 LTE로 전환되어 IP가 바뀌면 접속하던 서버와의 연결이 끊기고 다른 서버로 연결될 수 있음

    쿠키 기반

    • 정교한 타겟팅: IP가 같아도 브라우저에 저장된 쿠키가 다르면 서로 다른 서버로 분산 가능
    • 안정성: 네트워크 환경(IP)가 바뀌어도 쿠키만 유지된다면 기존 서버와의 세션 유지 가능

    단점

    • 쿠키 값을 읽고 써야하기 때문에 L7(애플리케이션) 계층의 로드 밸런서가 필요하며 부하가 약간 더 높다.
    • 쿠키에 서버 정보가 노출될 위험이 있어 암호화 설정이 권장

    따라서,

    • 일반적인 웹 서비스나 쇼핑몰 등에서는
      사용자의 IP 변경이 잦고 브라우저 단위 식별이 중요하므로 쿠키 기반 방식이 표준
    • 내부 사내망 등에서는
      사용자의 IP가 고정되어 있고 구조를 단순하게 가져가고 싶을 때는 IP 해시가 효율적일 수 있음

    고정 세션 방식이 필수가 아닌 대부분의 상황에서는 세션 클러스터링 방식(세션 복제, 세션 서버 분리)으로 여러 대의 서버에 세션 정보를 공유해 무상태(Stateless) 구조를 사용한다.
